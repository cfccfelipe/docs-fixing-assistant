## ### 1. The Core Agentic Loop (The "Heartbeat")

In a production system, you don't just "send a prompt." You manage a lifecycle.

- **The Signal:** You must monitor `stop_reason`.
    
    - `tool_use`: The model is asking for data. **Action:** Execute the code/API and feed it back.
        
    - `end_turn`: The model is done. **Action:** Show the result to the user.
        
- **The History:** Tool results are **appended** to the message history. The model "sees" the API output in the next turn and reasons: _"Okay, I have the user's ID now, I can proceed to the refund."_
    
- **Anti-Pattern Alert:** Never stop a loop just because the model says "I'm done" in plain text. Only trust the structured `stop_reason`.

## ### 2. Coordinator-Subagent (Hub-and-Spoke)

For complex tasks (like deep research or coding), a single agent gets "distracted" or loses focus.

- **Isolation is Key:** Subagents start with a **blank slate**. They do not know what the Coordinator knows unless the Coordinator explicitly passes that data in the prompt.
    
- **The "Task" Tool:** This is the specific mechanism used to spawn a subagent. The Coordinator must have this tool enabled in its `allowedTools`.
    
- **Parallelism:** A smart Coordinator doesn't call subagents one by one. It emits **multiple** Task calls in a single response to start three different research threads at once.

## ### 3. Deterministic vs. Probabilistic Enforcement

This is the most frequent area for "Senior" level exam questions.

|**Feature**|**Prompt-Based (Probabilistic)**|**Hook/Programmatic (Deterministic)**|
|---|---|---|
|**Example**|"Please verify user ID first."|A `PostToolUse` hook that blocks `refund` if `verify` isn't done.|
|**Reliability**|~90-95% (Model might skip it).|**100%** (Code-level gate).|
|**Best For**|Style, tone, general guidance.|Security, Financials, Identity, Compliance.|

> **Pro-Tip:** If a recruiter asks how you handle a $500 refund limit, your answer should be: _"I implement a tool-call interception hook to block the action via code, rather than just telling the AI not to do it."_

## ### 4. Advanced Decomposition (Avoiding "Attention Dilution")

When reviewing 14 files, a model's attention drops in the middle (the "Lost in the Middle" phenomenon).

- **Sequential Chaining:** Review File A → Review File B → Review File C.
    
- **Cross-File Pass:** Once individual reviews are done, run one final "Integration Pass" to check how File A and File C interact.
    
- **Adaptive Plans:** Don't give a 10-step list. Tell the agent: _"Map the codebase, then tell me the 3 most critical areas to test."_ Let the agent's findings dictate the next subtask.

|**Error Type**|**Agent Action**|
|---|---|
|**Transient** (Timeout)|**Retry** (The agent understands it might work in 5 seconds).|

## ### 1. Precision Over Vague Instructions

The biggest mistake in prompt engineering is using "soft" qualifiers.

- **The Anti-Pattern:** "Be very careful," "Only report high-confidence bugs," or "Be conservative." These are subjective and the AI interprets them inconsistently.
    
- **The Pro-Fix:** Use **Explicit Categorical Criteria**.
    
    - _Example:_ "Flag a comment **only if** the code performs a `POST` request but the comment says it is a `GET` request."
        
- **Trust Management:** If a specific type of check (like "variable naming") is generating too many false positives, **disable it entirely** until you can refine the prompt. Don't let "noise" ruin the team's trust in the tool.

## ### 2. Few-Shot Prompting (The "Show, Don't Tell" Rule)

Few-shot examples are the single most effective way to handle **ambiguity**.

- **Why it works:** It teaches the model the _reasoning_ behind a choice.
    
- **What to include:** 2–4 examples.
    
    - Show an **Ambiguous Case** and explain _why_ Tool A was picked over Tool B.
        
    - Show an **Acceptable Pattern** that looks like a bug but isn't (to reduce false positives).
        
    - Show the **Exact Output Format** you want (JSON structure, specific fields).

## ### 3. Guaranteed Structure: Tool Use & JSON Schemas

If you need JSON, don't just ask for it in the prompt. Use the **Tool Use API**.

- **The Mechanism:** By defining a tool with a JSON schema, you force the model to follow that schema. It eliminates syntax errors (like missing commas).
    
- **`tool_choice` Strategies:**
    
    - `"auto"`: Model decides (risky for structured tasks).
        
    - `"any"`: Model **must** call a tool. Use this for data extraction.
        
    - `{"type": "tool", "name": "..."}`: Forces a **specific** tool. Use this to ensure metadata is extracted before the analysis starts.
        
- **Schema Design:** Use `optional` (nullable) fields. If you make a field "required" but the data isn't in the document, the AI will **hallucinate** a value just to satisfy the schema.

    - **Pros:** 50% cheaper.
        
    - **Cons:** Up to 24-hour delay; **No multi-turn tool calling** (the AI can't call a tool, wait for the result, and then think again).

Recruiters love to ask: "When should the AI stop talking to the customer?"

- **Policy Gaps:** Escalate not just when things are "hard," but when the **policy is silent**. If a customer asks for a price match and your manual says nothing about it, the AI must not "guess"—it must escalate.
    
- **Multiple Matches:** If a search for "John Smith" returns three people, a bad agent guesses. A senior agent asks: _"I found three people with that name. Could you please provide your email or order number to help me find the right account?"_
    
- **Immediate Handoff:** If a customer says "Agent," "Human," or "Representative," you stop immediately. Do not try to "win them back" first; it only increases frustration.

### 3. Error Propagation (The "Chain of Command")

In a multi-agent system, a sub-agent failing shouldn't crash the whole project.

- **Structured Errors:** A sub-agent shouldn't just say "Failed." It should return:
    
    - **What it tried:** "I searched the PDF for 'Tax ID'."
        
    - **Why it failed:** "The file was encrypted."
        
    - **Partial results:** "I found the Name and Date, but not the ID."
        
- **Coordinator Recovery:** With this info, the Coordinator can decide: _"Okay, skip the ID but continue with the Name and Date,"_ rather than just giving up entirely.

### 4. Large Codebase Exploration (The "Scratchpad")

When using Claude Code on a 1,000-file repo, the "Context" fills up with search logs.

- **Scratchpads:** Have the agent write its findings into a temporary `.md` file. This acts as "external memory." If the session crashes or needs to be compacted, the agent can just read the scratchpad to remember where it was.
    
- **The `/compact` Command:** Periodically use this to clear out the "noise" of old search results while keeping the important summaries.

### 5. Provenance & Uncertainty

How do you handle two credible sources that contradict each other? (e.g., one report says 10% growth, another says 12%).

- **Source Mapping:** Never let the agent say "Growth is 11%." It must say "Source A reports 10% while Source B reports 12%."
    
- **Temporal Awareness:** Require sub-agents to include **dates** in their findings. Often, a "contradiction" is just one source being newer than the other.

## ### Step 1: The "Anti-Hallucination" Schema

When designing schemas for messy real-world data, **Required vs. Optional** is your most powerful lever.

**Tool Definition (`extract_security_incident`):**

- **`incident_type`**: Enum [`unauthorized_access`, `billing_spike`, `resource_deletion`, `other`].
    
- **`other_details`**: String (Optional). Used only if `incident_type` is `other`.
    
- **`impacted_region`**: String (Nullable).
    
- **`severity_score`**: Integer 1-10 (Required).
    

**The Test:** If you process a log that doesn't mention a region, a "Required" field forces the AI to guess (e.g., `us-east-1`). By making it **Nullable**, the AI returns `null`, preserving the truth.

## ### Step 2: The Validation-Retry Loop

Not all errors are equal. Your code needs to distinguish between a "fixable" AI mistake and a "missing data" reality.

|**Error Type**|**Status**|**Action**|
|---|---|---|
|**JSON Syntax**|Fixable|Retry: "You missed a closing bracket."|
|**Logic/Semantic**|Fixable|Retry: "The sub-totals ($50+$60) don't match the Total ($100)."|
|**Missing Info**|**Unfixable**|**Terminate**: "The source document simply does not contain a Tax ID. Do not retry."|

**Implementation:** Capture the `Pydantic` validation error and send it back to Claude: _"Your last extraction failed validation: 'severity_score' must be an integer, you provided 'High'. Please correct this using the original document."_

## ### Step 3: Few-Shot for "Format Shock"

AI often struggles when data moves from a table to a paragraph. You provide 2-3 "Exemplars" in your system prompt.

- **Example A (Narrative):** Show a paragraph where the Date is buried in a sentence.
    
- **Example B (Table):** Show a Markdown table where the Date is in a column.
    
- **Example C (Edge Case):** Show a document with _two_ dates and explain why we pick the "Effective Date" over the "Creation Date."

## ### Step 4: Batch Strategy & SLA Management

For a DevOps engineer, the **Message Batches API** is about balancing cost ($) vs. time.

- **Custom IDs:** Assign each document a `custom_id` (e.g., `doc_001`). If the batch returns with 98 successes and 2 failures, you use the ID to know exactly which files to re-submit.
    
- **Resubmission Logic:** If a failure is "Token Limit Exceeded," your script should **chunk** that specific document into two smaller parts before resubmitting.
    
- **SLA Calculation:** If your SLA is 30 hours, and the Batch API takes up to 24 hours, you must trigger your batch script **at least every 6 hours** to ensure you never miss a deadline.

## ### Step 5: Confidence Routing (The Human-in-the-Loop)

You don't need to review everything. You only review the "Grey Areas."

1. **Confidence Scores:** Ask the model to output a `confidence` float (0.0-1.0) for every field.
    
2. **The Threshold:** * **Score > 0.9:** Auto-approve and save to Database.
    
    - **Score < 0.9 OR `incident_type == "other"`:** Route to a human dashboard (like Amazon A2I or a custom UI).
        
3. **Stratified Sampling:** Even for "High Confidence" items, randomly pick 2% to be reviewed by a human. This ensures the model isn't "confidently wrong" about a new type of document.
## ### 1. The Coordinator Architecture (Hub-and-Spoke)

The **Coordinator** is the brain that doesn't do the work itself; it manages the "Specialists."

- **The "Task" Tool:** For this to work, your Coordinator's `allowedTools` must explicitly include `Task`. This is the "spawn" command for subagents.
    
- **Explicit Context Passing:** Remember, Subagent B does not know what Subagent A found unless the Coordinator reads A's output and injects it into B's prompt.
    
- **The Prompt:** Instead of "Research X," the Coordinator's prompt should be: _"Decompose 'Impact of AI in Manizales' into three sub-tasks: (1) Local Economy, (2) University Research, and (3) Infrastructure. Assign each to a specialized Search subagent."_

## ### 2. Parallel Execution (The Latency Win)

In a sequential loop, your total time is $T_1 + T_2 + T_3$. In parallel, it's $\max(T_1, T_2, T_3)$.

- **Implementation:** Claude can emit multiple tool calls in one turn. Your backend must support async execution of these `Task` calls.
    
- **DevOps Benefit:** This is exactly like parallelizing a Jenkins or GitHub Actions pipeline. You reduce the "Time to Result," which is a key metric for user experience.

## ### 4. Graceful Degradation (Error Propagation)

What happens when the "Web Search" agent hits a 403 Forbidden error or a Timeout?

- **The Anti-Pattern:** The whole system crashes or returns "No results found."
    
- **The Professional Pattern:** The subagent returns a **Partial Success Object**:
    
    JSON
    
    ```
    {
      "status": "partial_failure",
      "error_type": "timeout",
      "attempted_query": "AWS Cloud adoption 2026",
      "retrieved_data": ["Found 2 PDF titles, but could not download."]
    }
    ```
    
- **Coordinator Recovery:** The Coordinator sees this and adds a "Coverage Gap" note to the final report: _"Note: Data on AWS adoption is limited due to a source timeout; findings are based on metadata only."_

## ### 5. Handling Conflicts (The "Courtroom" Model)

When Source A says "10%" and Source B says "15%," the agent should act like a judge, not an editor.

- **Attribution Preservation:** Ensure the synthesis output uses **Inline Citations** (e.g., "[Source 1]").
    
- **Structure:**
    
    - **Verified Facts:** Points where all agents agree.
        
    - **Discrepancies:** Areas where data differs.
        
    - **Contextual Reasoning:** _"Source B is more recent (2026), which may account for the 5% increase over Source A (2024)."_
